IoT — Technical Documentation
Complete reference for the Odoo IoT system: architecture, modules, data flow, drivers, interfaces, communication protocols, debugging.

What is Odoo IoT?

Odoo IoT is a system that connects physical hardware devices with the Odoo application. It consists of two core components.

1.1 The IoT Box

A Raspberry Pi (or Virtual IoT on Windows) that runs a lightweight Odoo instance with the iot_drivers module. The IoT box:

  • Sits on the local network (LAN)
  • Connects to physical devices (printers, scales, payment terminals, etc.)
  • Acts as a proxy between those devices and the central Odoo database
  • Is itself a running Odoo server — it has its own Python environment, odoo.conf, and can host HTTP routes
1.2 The IoT App

An Odoo module (enterprise/iot) running on the central Odoo server that:

  • Manages IoT boxes and their devices
  • Communicates with IoT boxes via HTTP and WebSocket
  • Routes device actions from Odoo apps (POS, Reports, etc.) to the appropriate IoT box

2.2 Key Design Principles
  1. The IoT box is an Odoo server — It runs Odoo (Python), maintains its own config, hosts HTTP routes, and uses the same WSGI architecture as a regular Odoo instance.
  2. Thread-per-device pattern — Each detected device runs in its own daemon thread, allowing parallel device communication without blocking.
  3. Autonomous operation — Once configured, the IoT box runs independently. It can tolerate Odoo server downtime and reconnects automatically.
  4. RAM-disk safety — The filesystem is mostly read-only (RAM disk) to protect against SD card corruption from power loss. Only specific paths are persistent.
  5. No direct database access from devices — The IoT box does NOT connect directly to the Odoo database. All communication goes through HTTP/WebSocket APIs.

IoT Box Image Creation

4.1 What Gets Built

The IoT Box image is a Raspberry Pi OS (Debian-based, 32-bit ARM) SD card image containing:

  • Raspberry Pi OS Lite — minimal server edition, no desktop
  • Odoo — cloned from GitHub, sparse checkout including community/addons/iot_drivers and dependencies
  • nginx — reverse proxy routing port 8069 → 80, handling HTTPS
  • CUPS — Common UNIX Printing System for network/USB printer management
  • Chromium — for the display driver's browser (CLI mode)
  • hostapd + dnsmasq — Wi-Fi access point mode
  • Python dependencies — from requirements.txt
  • Systemd services: odoo.service, labwc.service, ramdisks.service, sos.service, odoo-led-manager.service, odoo-nginx.service, hostapd.service
4.2 Build Process
Step 1
Start QEMU/KVM
Emulates ARM processor to build RPi image on any x86 host. KVM with virtio networking, 8GB RAM, RPi OS image.
Step 2
Prepare Raspberry Pi OS Image
Download 2025-10-01-raspios-trixie-armhf-lite.img.xz, expand image by 2816 MiB, grow filesystem, e2fsck, resize2fs, label as "iotboxfs".
Step 3
Clone Odoo Repository (Sparse Checkout)
git clone -b {branch} --depth=1 --no-checkout. Includes only community/addons/iot_drivers/**, Odoo core files, and setup/iot_box_builder/**.
Step 4
Mount and Customize
Mount /dev/loop0p2 and /dev/loop0p1, copy qemu-arm-static for chroot, copy overwrite_before_init/* to /mnt.
Step 5
Chroot Initialization
/etc/init_image.sh runs inside chroot: apt-get install packages, pip install requirements, create users (pi, odoo), configure locales/keyboard/timezone, set up udev rules, configure nginx & CUPS, write iotbox_version file.
Step 6
Finalize
Unmount both partitions, zerofree (compress empty blocks), output: iotbox_YYYY.MM.DD.img.

Build command:

python3 setup/iot_box_builder/iot_box_builder.py \
  -i /path/to/raspi-os-image.img \
  -l pi \
  -d /tmp \
  -b 18.0 \
  -c /path/to/ca.pub

Flashing the image to SD card:

sudo dd if=/tmp/iotbox_YYYY.MM.DD.img of=/dev/sdX bs=4M status=progress

4.4 The RAM Disk Strategy
/
├── etc/                    (persistent - writable at boot)
├── var/log/odoo/          (RAM disk - lost on reboot)
├── tmp/                    (RAM disk)
├── home/pi/odoo/          (RAM disk - git clone is here!)
├── root_bypass_ramdisks/  (PERSISTENT - survives reboot)
│   ├── etc/ssl/           (certificates persist)
│   ├── var/lib/tailscale/ (tailscale state persists)
│   └── etc/netplan/       (Wi-Fi config persists)
└── /var/odoo/             (persistent - iotbox_version)

The ramdisks.service systemd service creates symbolic links at boot, redirecting /var/log, /tmp, and the Odoo home directory to RAM disks (tmpfs). This design:

  • Protects SD cards — Prevents write wear from log files and temp data
  • Survives power loss — Only critical config is persisted; most state is ephemeral
  • Enables fast boot — tmpfs is immediately available, no disk fsck needed

Connecting IoT Box to Odoo

5.1 Two Connection Methods
Method 1: Pairing Code (IoT Box Has Internet Access)

This is the standard method when the IoT box is connected to a network with internet access.

Step 1
Register with Proxy
IoT box calls _register_iot_box() → POST /odoo-enterprise/iot/connect-box with {serial_number}. Proxy returns {pairing_code, pairing_uuid}.
Step 2
Display Pairing Code
IoT box prints pairing code on connected receipt printer.
Step 3
User Enters Code in Odoo
User opens the Odoo IoT App wizard and enters the pairing code. Odoo calls iot-proxy.odoo.com with pairing code + DB URL + token.
Step 4
IoT Box Polls for Result
IoT box polls iot-proxy.odoo.com every ~15–40 seconds (exponential backoff). Pairing code valid for 2 hours.
Step 5
Credentials Saved & Restart
IoT box saves credentials to odoo.conf under [iot.box] section and restarts.
Step 6
Device Registration
On restart, Manager sends device list to /iot/setup, receives the WebSocket channel, and connects via WebSocket.
Method 2: Offline Pairing (Token-Based)

When the IoT box cannot reach iot-proxy.odoo.com but can reach the Odoo server directly.

Token format (v18+):

{url}?token={hex_token}&db_uuid={uuid}&enterprise_code={ec}&db_name={name}
Note Pre-v18 tokens used pipe-separated format (url|token|uuid|ec). v18+ uses URL query parameter format for better readability and easier copying.

User generates token in Odoo IoT App wizard → pastes token on IoT box homepage (http://{iot_box_ip}:8069) → IoT box calls /connect_to_server → parses URL → saves to odoo.conf → restarts → connects to database.

5.2 Certificate Management (HTTPS Setup)
Step 1
Check Certificate Validity
certificate.ensure_validity() checks if /etc/ssl/certs/nginx-cert.crt exists, is valid (not expired, not a temp cert), and has >10 days remaining.
Step 2
Request New Certificate
If invalid: POST https://www.odoo.com/odoo-enterprise/iot/x509 with {db_uuid, enterprise_code}.
Step 3
Install Certificate
Write certificate to /etc/ssl/certs/nginx-cert.crt, /etc/ssl/private/nginx-cert.key, and persistent paths under /root_bypass_ramdisks/.
Step 4
Reload nginx & Notify Database
Run nginx -s reload. POST /iot/box/update_certificate_status with expiry date.
Design Detail The certificate is a Let's Encrypt certificate issued through Odoo.com as an intermediary. The IoT box does NOT run its own ACME client (certbot).

Data Flow: Entry Point to Device

6.1 The Thread Architecture

When Odoo starts on the IoT box, iot_drivers/__init__.py executes and starts all interface threads. The complete thread structure:

Main Process (Odoo WSGI)
├── Manager Thread (daemon)
│   ├── Checks device changes every 3 seconds
│   ├── Manages scheduled tasks (daily cert check, weekly git update)
│   └── Runs WebSocketClient
├── ConnectionManager Thread (daemon)
│   ├── Polls iot-proxy.odoo.com for pairing result
│   └── Manages pairing state machine
├── Interface Threads (daemon) — one per interface type
│   ├── USBInterface: polls USB every 3 seconds
│   ├── SerialInterface: polls serial ports every 3 seconds
│   ├── BluetoothInterface: polls BT every 3 seconds
│   └── ...
├── Driver Threads (daemon) — one per connected device
│   ├── PrinterDriver: polls CUPS job status every 1 second
│   ├── ScaleDriver: reads serial port every 0.2 seconds
│   ├── DisplayDriver: refreshes browser every 60 seconds
│   └── PaymentTerminal: processes action queue
├── WebSocketClient Thread (daemon)
│   └── Receives/sends WebSocket messages
└── Odoo HTTP Workers (request handling)
    └── Serves IoT box homepage and driver routes
6.2 Complete Print Flow (Detailed)
Step 1 — POS User Clicks "Print Receipt"
RPC to IoT Box
POS JavaScript calls rpc('/hw_drivers/action', { session_id, device_identifier: 'usb_04b3:400c', data: { action: 'print_receipt', receipt: 'base64...' } }). Odoo's HTTP system proxies /hw_drivers/* to the IoT box.
Step 2
Message Sent via WebSocket
iot.channel.send_message() pushes to bus.bus._sendone('iot_channel-xxx', 'iot_action', message). Odoo WebSocket (longpolling) pushes to all subscribers.
Step 3
IoT Box Receives Message
WebSocketClient.on_message() parses the message, filters by iot_identifier, calls communication.handle_message('iot_action', 'ws', **payload).
Step 4
Message Routed to Device Driver
main.iot_devices['usb_04b3:400c'].action(payload). PrinterDriver:
1. Base64 decode receipt image
2. Convert to grayscale, then 1-bit (dithered)
3. Choose ESC/POS format by printer model (Star TSP100 / Epson TM-xxx / Epson TM-U220)
4. Add cut command → print via CUPS IPP.
Step 5
Status Polling
PrinterDriver polls CUPS job status every 1 second. On completion: send_status('success') → event_manager.device_changed() → HTTP POST to /iot/box/send_websocket.
Step 6
Response Flowed Back to POS
Odoo /iot/box/send_websocket validates response → pushes operation_confirmation via bus.bus → POS JavaScript receives → UI shows "Payment Complete, Receipt Printed".

How Other Modules Use the IoT System

7.1 Point of Sale (POS)

The POS module has deep IoT integration through several mechanisms:

7.1.1 Display (Customer-Facing)
  • POS connects to the IoT box's display via the PoS Self app
  • The DisplayDriver runs a Chromium browser showing the POS order summary
  • Kiosk mode (/pos-self/{pos_id}?access_token=...): Full-screen self-service POS
  • Customer display mode (/pos_customer_display/{pos_id}/{token}): Customer-facing screen alongside cashier POS
7.1.2 Printer
  • POS renders receipts as SVG/images in the browser
  • Sends print action via WebSocket → IoT box → CUPS → Printer
  • Supports Star TSP100 (Star commands) and Epson TM/TM-U (ESC/POS) receipt printers
  • Cashbox: Sends pulse signal via ESC/POS drawer kick command
  • Label printers: ZPL-compatible label printers for product labels
7.1.3 Payment Terminal
  • POS initiates payment through the IoT box
  • Supports Worldline/Six and Ingenico payment terminals
  • IoT box uses ctypes to call native .so libraries provided by the payment terminal vendor
  • Handles: payments, refunds, cancellations, balance inquiries
  • Real-time status updates (waiting for card, processing, approved, declined)
7.1.4 Scale
  • POS JavaScript subscribes to scale device via longpolling at /hw_drivers/event
  • When weight changes, the event is delivered to the waiting session
  • POS automatically updates the line item weight in the order
7.2 Reports (ir.actions.report)

ir.actions.report can be linked to IoT printer devices via the device_ids field. Print flow for reports:

  1. User clicks "Print" in Odoo
  2. Report renders to PDF server-side
  3. PDF is sent as document action data to the IoT box
  4. IoT box receives via WebSocket → PrinterDriver._action_default()
  5. PrinterDriver prints via CUPS
7.3 Custom Handlers Distribution

Odoo can push custom device drivers to the IoT box without rebuilding the image:

1
Enable Custom Handlers
User enables "Custom Handlers" on iot.box record in Odoo UI.
2
WebSocket Notification
Odoo sends toggle_custom_handlers message to IoT box via WebSocket.
3
Download Handlers ZIP
IoT box calls POST /iot/get_handlers. Odoo searches all installed modules for iot_handlers/ directories, excludes platform-incompatible files, packages into ZIP with ETag.
4
Extract and Restart
IoT box extracts to odoo/addons/iot_drivers/iot_handlers/, then restarts. On restart, new driver classes register themselves automatically.

This allows: hardware vendors to distribute drivers as Odoo modules, custom integrations without rebuilding the IoT image, and per-customer specific drivers.

Communication: HTTP vs WebSocket

8.1 The Three Communication Paths
Path Direction Protocol Use Case
IoT Box → Odoo (HTTP) Unidirectional HTTP POST Registration, device list updates, certificate status, log upload
WebSocket Bidirectional WebSocket Real-time commands: print, pay, display, scale reading
Odoo → IoT Box (HTTP) Unidirectional HTTP IoT box homepage, status page, logs page, config APIs
8.2 HTTP Communication (IoT Box → Odoo)

All HTTP calls from the IoT box to Odoo are decorated with @require_db, ensuring calls are only made when the IoT box has an IP address, is not in access point mode, and is connected to a database.

All HTTP requests include headers:

'User-Agent': 'OdooIoTBox/1.0'
'X-Odoo-Database': db_name
8.3 WebSocket Communication

The IoT box maintains a persistent WebSocket connection to the Odoo server's Bus system. On open, it subscribes to the IoT channel with its token and identifier. On message, it filters by iot_identifier, routes to the appropriate handler, and sends acknowledgements back via HTTP.

Reconnection strategy: WebSocket library handles reconnection with run_forever(reconnect=10). On disconnect, tries to reconnect every 10 seconds. The Manager thread also monitors and can restart the WebSocket client.

8.4 WebSocket Authentication

Method 1 — Session Cookie (pre-v18, still used): The WebSocketClient obtains a session via GET /web/login and sends the session cookie in the WebSocket headers.

Method 2 — IoT Token (v18+): The IoT token is sent in the WebSocket subscribe message. On the server, IrWebsocket._subscribe() validates the token against the iot.box record and adds the IoT channel to the subscription. Provides database-independent authentication.

8.5 Longpolling (Event Waiting)

For devices that need to wait for responses (scale reading, payment confirmation), Odoo uses longpolling via POST /hw_drivers/event. The event manager checks for recent events (last 5 seconds). If no matching event is found, it blocks via event.wait(50) for up to 50 seconds. When a device reports data, all waiting sessions for that device are notified. This longpolling mechanism is essential for real-time POS interactions.

    # Driver.__init_subclass__() → drivers.append(PrinterDriver_L)

Step 2: Interface Class Definition
    class USBInterface(Interface):
        connection_type = 'usb'
        allow_unsupported = True
        def get_devices(self): return usb_devices

Step 3: IoT Box Startup
    for interface_thread in main.interfaces.values():
        interface_thread().start()

Step 4: Interface.run() Loop
    while True:
        devices = self.get_devices()
        self.update_iot_devices(devices)

Step 5: Device Matching
    def add_device(identifier, device):
        supported_driver = next(
            (d for d in self.drivers if d.supported(device)), None)
        if supported_driver:
            d = supported_driver(identifier, device)
            iot_devices[identifier] = d
            d.start()  # Start driver thread
9.2 The Interface Class

Each interface subclass runs in a loop, polling for devices of its connection type every _loop_delay seconds (default: 3). It maintains a set of _detected_devices and calls add_device() for new devices and remove_device() for disconnected ones. The loop exits only if no drivers are registered for the connection type.

9.3 The Driver Class

Each driver runs as a daemon thread, assigned to a specific device. Key properties:

  • connection_type — must match the interface's type
  • priority — higher priority drivers are checked first during device matching
  • _actions — dict mapping action names to handler methods
  • _recent_action_ids — LRU(256) cache for action deduplication
  • data — holds current device state: {'value': '', 'result': ''}

The action(data) method dispatches incoming actions, deduplicates via action_unique_id, calls the appropriate handler, and notifies Odoo (except for printers and payment terminals that handle their own event lifecycle).

Limitations and Why They Exist

11.1 Device Identifier Persistence

Problem: Device identifiers (especially USB) can change across reboots, causing Odoo to treat the same physical device as a new device.

Root cause: USB devices are identified only by vid:pid. The USB specification does not mandate unique serial numbers for all device classes. Example: two identical receipt printers — before reboot: usb_04b3:400c and usb_04b3:400c_2 — after reboot: may swap.

Why this design: Using only vid:pid is the most portable approach that works across all USB devices without requiring vendor cooperation. Implementing serial number tracking would require device-specific handling for each vendor.

11.2 Network Dependency for Pairing

Problem: Initial pairing requires the IoT box to reach iot-proxy.odoo.com.

Why: The proxy prevents unauthorized IoT boxes from connecting to your database, provides a simple pairing code UX, and acts as a relay when the IoT box and Odoo server are on different networks.

Workaround: Offline pairing (token-based method) allows connecting when the proxy is unreachable but the Odoo server is reachable on the LAN.

11.3 USB Device Polling (Not Event-Based)

Problem: USB device connect/disconnect is detected via polling every 3 seconds, not via kernel uevent.

Root cause: The original pyudev library (which uses kernel uevent) was difficult to install and maintain on the Raspberry Pi image. The usb library (pyusb) requires polling.

Impact: Maximum 3-second delay in detecting USB device connect/disconnect. Acceptable because for most IoT use cases (printers, scales), devices are connected at startup and stay connected.

11.4 CUPS Dependency for Linux Printing

Problem: Linux IoT box relies entirely on CUPS for printer communication.

Why: CUPS provides a standardized IPP interface that works with most network printers and is the standard Linux printing architecture.

Limitations: Some USB printers require vendor-specific drivers not in CUPS; CUPS's IPP may not support all printer features; USB printer identifier approach conflicts with CUPS enumeration.

11.5 Single WebSocket Channel Per Company

Problem: All IoT boxes from the same company share one WebSocket channel.

Why: Odoo's bus system is designed for database-level channels. The IoT channel is created once (iot.ws_channel config parameter) and all IoT boxes subscribe to it.

Impact: All boxes receive all messages (filtered by iot_identifier). A misbehaving IoT box won't break others' communication. No per-box channel isolation. This is acceptable because the iot_identifier filter ensures each box only processes its own messages.

11.6 Certificate Auto-Renewal Limitations

Problem: Certificates are renewed via Odoo.com intermediary, not a continuous ACME process.

Advantages: Simpler implementation, works behind firewalls, no ACME port 80 requirement.

Limitations: Depends on Odoo.com's certificate service; maximum 5 retry attempts with 95-second timeout per attempt; certificates are not auto-renewed at 30 days (only when <10 days remain).

11.7 RAM Disk Persistence Trade-offs

What persists (under /root_bypass_ramdisks/): Odoo config, SSL certificates, Wi-Fi/netplan configuration, Tailscale state, Odoo version file.

What is lost on reboot: All log files, temporary files, git repository state (re-cloned on boot), custom handler files (re-downloaded from Odoo DB if enabled).

11.8 Raspberry Pi Only (Physical)

Why: Image is built specifically for Raspberry Pi OS (Debian ARM); device tree serial numbers used for IoT box identifier; hardware-specific services (LED manager, hardware watchdog); ARM architecture assumed throughout.

Why acceptable: Raspberry Pi is affordable, widely available, well-documented, and has excellent Linux support.

11.9 Handler Architecture Limitations

Problem: Custom handlers are wiped on git checkout. When the IoT box runs upgrade.check_git_branch(), git clean -dfx removes all untracked files including custom handlers.

Why this design exists: Custom handlers are treated as part of the Odoo module ecosystem. They're downloaded from the Odoo database (which stores them as installed module assets), not as standalone files.

Workaround: Enable "Custom Handlers" on the iot.box record. After git clean, the Manager downloads and re-installs handlers automatically on the next restart.

11.10 Platform-Specific Driver Files

Why: Different operating systems have different driver libraries (CUPS vs. Win32print), USB APIs (pyusb vs. WinUSB), and browser execution approaches.

Why acceptable: The platform split is minimal and necessary. Most device interaction code (serial protocols, payment terminal ctypes, ESC/POS formatting) is in platform-independent base classes. Only the hardware I/O layer is platform-specific.

11.11 No Real Multi-Threading Safety for Some Resources

Problem: Some shared resources (like the CUPS connection) use simple locks rather than sophisticated concurrency control. PrinterDriver uses a cups_lock (threading.Lock) for all CUPS operations.

Why acceptable: The lock ensures thread-safe access to the CUPS connection object. For most IoT use cases (1–2 printers), this is sufficient. High-throughput scenarios might need connection pooling.

IoT Subscription & Certificate Lifecycle

12.1 What Does "Subscription" Mean for IoT?

There is no subscription button, no expiry date field, no "renew now" popup in the IoT module. The subscription is entirely invisible — it works through the SSL certificate.

Here is the simple idea:

  • Every IoT box needs a valid HTTPS certificate to communicate with the Odoo server
  • The IoT box cannot generate this certificate on its own — it must ask Odoo.com to issue one
  • Odoo.com only issues a certificate if your database has a valid Enterprise license
  • The certificate expires after ~60 days, so it must be renewed regularly
  • If the license is not valid, renewal fails, the certificate expires, and HTTPS breaks — the IoT box effectively stops working with Odoo

In short: the subscription IS the certificate. No valid license = no valid certificate = no communication.

Question Answer
What two things does the IoT box need? db_uuid (database ID) and enterprise_code (Enterprise license)
Where are they stored on the IoT box? In /etc/odoo/odoo.conf under the [iot.box] section
Where do they come from? Saved automatically when the IoT box is paired with an Odoo database
Who checks the license? Odoo.com — when the IoT box asks for a certificate, odoo.com checks the license before issuing
Is there code on the IoT box checking subscription? No. The IoT box never checks "is my subscription active?" It only asks for certificates. Odoo.com decides whether to give one.
12.2 How It Actually Works — The Full Flow

Here is how the subscription + certificate system works from the moment you build the IoT image to daily operation. Each step tells you what happens and where in the code it is handled.

Step 1
Image Build — A Temporary Certificate Is Included

When someone builds the IoT box image (the .img file you flash to an SD card), they include a temporary self-signed certificate. This is NOT a real certificate — it's just a placeholder so nginx can start.

Where:

  • Build script: setup/iot_box_builder/iot_box_builder.py
  • Temp cert file: setup/iot_box_builder/overwrite_before_init/etc/ssl/certs/nginx-cert.crt
  • Temp key file: setup/iot_box_builder/overwrite_before_init/etc/ssl/private/nginx-cert.key

What you'll see if you inspect the cert: The Common Name (CN) is OdooTempIoTBoxCertificate. It's valid for 100 years (2018–2118) — clearly not a real cert. The IoT code specifically looks for this name to know "this is the placeholder, I need a real one."

At this point: The IoT box is just a blank image. It has no database, no license, no real certificate.

Step 2
First Boot — IoT Box Runs with the Temporary Certificate

When you plug in the IoT box and it boots up, nginx starts using the temp certificate. The IoT box shows a pairing code on its screen/printer. But it cannot download a real certificate yet because it has no database credentials.

Where this is handled:

  • Main entry point: community/addons/iot_drivers/main.py → Manager.run()
  • Certificate check: community/addons/iot_drivers/tools/certificate.py → ensure_validity()
  • The check sees the CN is OdooTempIoTBoxCertificate → knows it's invalid
  • It tries to download a real cert, but db_uuid is empty (not paired) → gives up

What you'll notice: The IoT box homepage loads, but browsers show a security warning (self-signed cert). Everything works locally (devices, printing), but it cannot talk to Odoo over HTTPS.

Step 3
Pairing — Database Credentials Are Saved

The user pairs the IoT box with an Odoo database using the pairing code (or offline token). This is where db_uuid and enterprise_code get saved to the IoT box.

Where this is handled:

  • Pairing starts from Odoo app: enterprise/iot/wizard/add_iot_box.py → calls iot-proxy
  • IoT box receives credentials: community/addons/iot_drivers/connection_manager.py → _connect_to_server()
  • Credentials saved: community/addons/iot_drivers/tools/helpers.py → save_conf_server(url, token, db_uuid, enterprise_code)
  • IoT box restarts to apply the new config

What's saved in odoo.conf:

[iot.box]
remote_server = https://my-odoo.example.com
token = a1b2c3d4...
db_uuid = abc-123-...
enterprise_code = ENT-456-...

At this point: The IoT box knows which database it belongs to, but still has the temp certificate.

Step 4
After Restart — Real Certificate Downloaded from Odoo.com

After the restart, the certificate check runs again. This time, db_uuid and enterprise_code exist — so the IoT box asks Odoo.com for a real certificate.

Where this is handled:

  • Certificate download: community/addons/iot_drivers/tools/certificate.py → download_odoo_certificate()
  • Request sent to: https://www.odoo.com/odoo-enterprise/iot/x509
  • Odoo.com checks: "Does this db_uuid have a valid Enterprise license?"
  • If YES → issues a real Let's Encrypt certificate
  • If NO → returns an error, IoT box keeps the temp certificate

Where the cert is saved:

Path Purpose Survives Reboot?
/etc/ssl/certs/nginx-cert.crt RAM disk — nginx reads from here No (but re-synced from SD at boot)
/root_bypass_ramdisks/etc/ssl/certs/nginx-cert.crt Real SD card — persistent storage Yes
/etc/ssl/private/nginx-cert.key Private key (RAM) No
/root_bypass_ramdisks/etc/ssl/private/nginx-cert.key Private key (persistent) Yes

The cert is written to both locations because: RAM disk is what nginx reads right now, but the SD card is what survives reboots. The IoT box uses setup_ramdisks.sh at boot to copy from SD → RAM.

After this: The IoT box has a real certificate. HTTPS works. The IoT box is fully connected to Odoo. All features work.

Step 5
Daily Auto-Renewal — Certificate Stays Fresh

The real certificate expires after about 60 days. To prevent this, the IoT box checks every day at midnight whether the certificate needs renewal.

Where this is handled:

  • Scheduler: community/addons/iot_drivers/main.py → schedule.every().day.at("00:00").do(certificate.ensure_validity)
  • Check logic: community/addons/iot_drivers/tools/certificate.py → get_certificate_end_date()
  • If less than 10 days until expiry → calls download_odoo_certificate() again
  • Odoo.com validates the license again → issues a new cert → old cert is replaced
  • Also runs on every IoT box boot (not just midnight)

Retry behavior: If the download fails (network issue, odoo.com down), it retries up to 5 times with a 95-second timeout each. If all 5 fail, it waits until the next day's check.

12.4 How the IoT Box Knows It's Connected

Print Receipt: Complete Flow (Every Step)

13.1 Overview — Two Communication Paths

When POS sends a print receipt command, there are two possible paths depending on the connection method:

Path Protocol Used When
Path A: Longpolling (Direct HTTP) HTTP POST from browser → IoT box Default path. Browser sends directly to IoT box IP.
Path B: WebSocket (Via Odoo Bus) Odoo Bus → WebSocket → IoT box Fallback when longpolling fails. Message routed through Odoo server.

Both paths eventually reach the same handler on the IoT box. The response always comes back via the same mechanism that triggered the action.

    }

Step 7
Driver.action() — Action Dispatcher

Dispatches to the specific action handler. Checks for duplicate actions via LRU cache:

# community/addons/iot_drivers/driver.py
class Driver(Thread):
    def action(self, data):
        action = data.get('action', '')          # 'print_receipt'
        action_unique_id = data.get('action_unique_id')

        # Deduplication: skip if this action was already processed
        if action_unique_id:
            if action_unique_id in self._recent_action_ids:
                return {'status': 'duplicate'}
            self._recent_action_ids[action_unique_id] = action_unique_id

        session_id = data.get('session_id')
        try:
            response = {'status': 'success', 'result': self._actions[action](data), 'session_id': session_id}
            # Printers return "pending" — they handle their own event lifecycle
            if self.device_type in ["printer", "payment"]:
                response['result'] = "pending"
        except Exception as e:
            response = {'status': 'error', 'result': str(e), 'session_id': session_id}

        return response
Step 8
PrinterDriverBase.print_receipt() — Image Processing

Decodes the base64 receipt image and converts it to a printer-compatible format:

# community/addons/iot_drivers/iot_handlers/drivers/printer_driver_base.py
class PrinterDriverBase(Driver, ABC):
    def print_receipt(self, data):
        receipt = b64decode(data['receipt'])      # Decode base64 image
        im = Image.open(io.BytesIO(receipt))     # Open as PIL Image

        # Convert to greyscale then to black and white
        im = im.convert("L")        # 8-bit greyscale
        im = ImageOps.invert(im)    # Invert colors (receipt printers use black-on-white)
        im = im.convert("1")        # 1-bit dithered black and white

        # Choose format based on printer protocol (star or escpos)
        print_command = getattr(self, 'format_%s' % self.receipt_protocol)(im)

        # Send to CUPS
        self.print_raw(print_command, action_unique_id=data.get("action_unique_id"))
Step 9
format_escpos() or format_star() — Protocol Encoding

Converts the PIL image to ESC/POS or Star raster commands. For ESC/POS (Epson):

# printer_driver_base.py
@classmethod
def format_escpos(cls, im):
    # Determines printing mode: 'raster' (GS v 0) or 'column' (ESC *)
    # Raster mode (default):
    width = int((im.width + 7) / 8)
    raster_send = b'\x1d\x76\x30\x00'  # GS v 0 command
    raster_data = b''
    dots = im.tobytes()
    while len(dots):
        im_slice = dots[:width * 255]
        slice_height = int(len(im_slice) / width)
        raster_data += raster_send + width.to_bytes(2, 'little') + slice_height.to_bytes(2, 'little') + im_slice
        dots = dots[width * 255:]
    return raster_data + cls.RECEIPT_PRINTER_COMMANDS['escpos']['cut']
Step 10
PrinterDriver.print_raw() — Send to CUPS via IPP

Sends the formatted data to CUPS (Common UNIX Printing System) which manages the actual printing:

# community/addons/iot_drivers/iot_handlers/drivers/printer_driver_L.py
class PrinterDriver(PrinterDriverBase):
    def print_raw(self, data, action_unique_id=None, duplex=True):
        with self.cups_lock:  # Thread-safe CUPS access
            options = {'document-format': CUPS_FORMAT_AUTO}
            if not duplex:
                options["sides"] = "one-sided"

            # Create CUPS job
            job_id = self.conn.createJob(self.device_identifier, 'Odoo print job', options)

            # Send document data
            self.conn.startDocument(self.device_identifier, job_id, 'Odoo print job', CUPS_FORMAT_AUTO, 1)
            self.conn.writeRequestData(data, len(data))
            self.conn.finishDocument(self.device_identifier)

        # Track job for status polling
        self.job_ids.append(job_id)
        if action_unique_id:
            self.job_action_ids[job_id] = action_unique_id

The CUPS daemon then sends the IPP commands to the printer via USB or network.

On the IoT box, DriverController.event() handles this by checking for recent events and waiting for new ones:

# community/addons/iot_drivers/controllers/driver.py
@route.iot_route(['/iot_drivers/event', '/hw_drivers/event'], type='jsonrpc', cors='*', csrf=False)
def event(self, listener):
    req = event_manager.add_request(listener)

    # Check for events from the last 5 seconds
    oldest_time = time.time() - 5
    for event in list(event_manager.events):
        if event['time'] < oldest_time:
            del event_manager.events[0]
            continue
        if event['device_identifier'] in listener['devices'] and event['time'] > listener['last_event']:
            return event  # ← Return immediately if event found

    # Wait for new event (up to 50 seconds)
    if req['event'].wait(50):
        req['event'].clear()
        return req['result']

# printer_driver_L.py
def _check_job_status(self, job_id):
    with self.cups_lock:
        job = self.conn.getJobAttributes(job_id, requested_attributes=[
            'job-state', 'job-state-reasons', 'job-printer-state-message', 'time-at-creation'
        ])
        job_state = job['job-state']

        if job_state == IPP_JOB_COMPLETED:        # Job finished successfully
            self.job_ids.remove(job_id)
            self.job_action_ids.pop(job_id, None)
            self.send_status(status='success')    # ← TRIGGERS EVENT BACK TO ODOO

        elif job['time-at-creation'] + self.job_timeout_seconds < time.time():
            self._cancel_job_with_error(job_id, 'ERROR_TIMEOUT')

        elif job_state == IPP_JOB_PROCESSING and 'printer is unreachable' in job.get('job-printer-state-message', ''):
            self._cancel_job_with_error(job_id, 'ERROR_UNREACHABLE')

        elif job_state not in [IPP_JOB_PROCESSING, IPP_JOB_PENDING]:
            self._cancel_job_with_error(job_id, 'ERROR_UNKNOWN')

# community/addons/iot_drivers/event_manager.py
class EventManager:
    def device_changed(self, device, data=None):
        event = {
            **device.data,                      # {'status': 'success', 'message': None}
            'device_identifier': device.device_identifier,
            'time': time.time(),
        }

        # 1. Send to Odoo database via HTTP POST (for WebSocket path)
        send_to_controller({
            **event,
            'iot_box_identifier': IOT_IDENTIFIER,
        })

        # 2. Wake up any waiting longpolling sessions (for longpolling path)
        self.events.append(event)
        for session in self.sessions:
            session_devices = self.sessions[session]['devices']
            if (
                any(d in [device.device_identifier, device.device_type] for d in session_devices)
                and not self.sessions[session]['event'].is_set()
            ):
                self.sessions[session]['result'] = event
                self.sessions[session]['event'].set()  # ← Wakes up the waiting thread

For the longpolling path, event.set() wakes up the req['event'].wait(50) in DriverController.event(), which returns the event to the POS.
